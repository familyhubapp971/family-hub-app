The Community Administrator
Runs a mosque youth programme, homeschool co-op, church family ministry, or NGO. Manages 20–500 families on behalf of an organisation. Buys the white-label tier ($299–$3,499/yr).
KEY MOTIVATION
"My families deserve better than a WhatsApp broadcast and a Google Form. I want a branded digital home for our community."
KEY OBJECTION
"I need to take this to our board. Can I see a demo and get a one-pager? We also need to know about data privacy for our children."
BUYING CYCLE
4–8 weeks. Board approval. Needs demo + reference customer + one-pager. PO/invoice billing path required.

Gaps & Open Questions — Persona 2 has the most backlog gaps
S1 · COMMUNITY LANDING PAGE
No Jira ticket. No feature doc. No design. The community/org marketing page is entirely absent — the current familyhub.app landing targets individual families only. Needs a dedicated route (/community) with org-specific messaging, testimonials, and pricing table.
S2 · TALK-TO-SALES LEAD FORM + CRM HANDOFF
No ticket. No spec. Lead capture form doesn't exist. Also undefined: where does the lead land? Which CRM? Who owns the follow-up? No sales pipeline tooling scoped yet. Required before any community org can enter the funnel.
S3 · SALES-LED ONBOARDING INTAKE (MULTI-STEP)
No ticket. White-label onboarding flow (logo + colour + subdomain + billing contact) has no spec, no Jira story, no AC. The existing tenant-creation path (FHS-150s) is self-serve consumer only — it does not accommodate org-level setup with branding.
S5 · WHITE-LABEL THEME PICKER
No ticket. Brand colour injection, logo upload, and font selection are not implemented or specced. Custom subdomain exists in concept (ADR-0002) but no per-tenant colour token system is designed. Needs ADR covering CSS variable injection strategy.
S6 · BULK CSV INVITE
No ticket. Bulk family invite via CSV upload is not specced. Current invite path (FHS-193 area) is one-at-a-time. Needs: CSV parser, validation UI, preview table, row-level error display, and batch invite job. Pricing tiers reference this as included in 101–500 families plan.
S7 · ORG ANALYTICS DASHBOARD
No ticket. No data model. No metrics defined. Org-level aggregate analytics (active families, engagement %, churn, feature usage) require: event pipeline, metrics aggregation job, and admin-only API surface. None of these are in the current backlog.
S8 · MULTI-ADMIN + OWNERSHIP TRANSFER
No ticket. Current role model supports family-level roles only (per ADR-0009). Org-level co-admin roles, permission matrices, and ownership transfer require a separate roles layer. No ADR exists for org-level RBAC. Risk: org admins leave, no succession path.
S9 · PO / PURCHASE-ORDER BILLING PATH
ADR-0004 (Stripe) exists but does not address PO/invoice billing. Many religious orgs and NGOs can only pay via purchase order with 30-day net terms. Stripe supports this via manual invoices — but the flow (request → proforma → PO upload → payment confirmation) is unspecced.
S10 · SUSPENDED ORG / GRACE PERIOD STATE
No ticket. What happens when an org's subscription lapses? No grace period policy defined. No locked-state UI. No member communication flow. Risk: 142 families lose access silently. Needs: grace period duration policy, warning emails, progressive feature lockout, data export window.
OPEN QUESTION — MULTI-TENANT ADMIN DASHBOARD (S4)
S4 is partially implied by existing multi-tenancy work (FHS-150s, ADR-0001/0002) but no dedicated admin dashboard feature doc exists. Who sees the family list? Can an org admin see inside a family's private data? RLS boundary and admin data-access policy undefined.
