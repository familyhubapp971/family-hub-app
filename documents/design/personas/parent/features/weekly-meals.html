Legacy Reference

How it works today in family-hub (legacy, single-tenant)

01

WeeklyMeals.tsx renders 7 day-cards, each with breakfast / lunch / dinner slots. Clicking a slot opens an inline input; blur or Enter fires a fire-and-forget api.meals.upsert(day, slot, {mealName}) call.

02

MealsSection.tsx wraps WeeklyMeals and a HealthyHabits tab in a two-tab switcher (Home Meals / Healthy Habits).

03

weekly_meals table: day_of_week INT, meal_type TEXT, meal_name TEXT, notes TEXT. Unique on (day_of_week, meal_type). No week scoping — single-tenant, current week only.

04

weekly_snacks table: snack_name, snack_time ('morning'|'after_school'), sort_order. Rendered in a separate card below the grid; delete-on-hover.

05

No shopping list aggregation, no recipe storage, no ingredient tracking, no member preference/allergy logic, no meal history — all are out of scope in legacy.

06

New SaaS architecture must add tenant_id + week_id foreign keys, full RLS policies, and a recipes table — all absent in legacy.

RLS: every table (weekly_meals, recipes, ingredients, shopping_items, snack_inventory) carries tenant_id. Postgres RLS policies enforce isolation — no cross-tenant reads possible. Pattern per ADR 0001.
