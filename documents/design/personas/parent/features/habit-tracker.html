Habit Tracker

Under the new SaaS architecture — how the coordinating parent uses it day-to-day

The Habit Tracker gives the coordinating parent a single weekly grid view across every family member's habits. Streaks, sticker rewards, and a week-close ritual make progress visible and intrinsically motivating for children while keeping the parent's management overhead near zero.

How it works in the legacy family-hub today

Reference only
  • Single-family, single-tenant — one Postgres instance; no tenant_id on any table. All members share one namespace.
  • Week model — the weeks table tracks ISO week-number + year + start-date + finalization state; stickers earned carry over as cash via carried_over_stickers.
  • Habit × day grid — habits are global (not per-member); completion is recorded in habit_stickers keyed by (habit_id, week_id, day); sticker type (gold-star / heart / sparkles / trophy) is stored as a string.
  • Week-close ritual — admin finalizes the week via is_finalized = true; week_actions records the disposition: claim reward, cash out, save to piggy bank, or invest in a bonus habit. Results in AED are computed at 0.5 AED per sticker.
  • HabitTracker.tsx — React component with lazy-loaded AnalyticsView tab, optimistic day-cell updates, role-based day-lock (non-admin can only mark today), and an edit/delete dialog for admin. Icons and colors are stored as strings (star / heart / sparkles / award / zap / checkcircle) mapped to Lucide elements at render time.
  • Gaps vs new SaaS — no tenant_id, no per-member habit ownership, no REST API layer (component calls api.* helpers directly), no push notifications, no multi-tenant RLS, no Supabase auth integration.

Architecture Mapping — Key User Actions

Where each action lands in the new SaaS stack · tenant_id propagates on every write

User Action WEB (React + Vite) API (Hono + Zod) DATA (Postgres + RLS)
Mark habit done
FHS-3 (schema)
DayCell component → onToggle(habitId, dayIdx) → optimistic update in useHabits() hook
POST /api/habits/:id/actions · Zod body: {day, sticker, weekId} · tenant middleware sets req.tenantId
INSERT INTO habit_stickers (tenant_id, habit_id, week_id, day, sticker) · RLS enforces tenant isolation
Create habit
FHS-backlog (CRUD API)
HabitForm modal → createHabit() mutation (React Query / SWR) · optimistic insert to local list
POST /api/habits · Zod body: {name, cadence, targetCount, color, icon, memberIds[]}
INSERT INTO habits (tenant_id, name, cadence, target_count, color, icon) + habit_members junction rows
View member habits
FHS-backlog (member filter)
MemberHabitsPage → useHabits({memberId}) → renders HabitRow list with avatar
GET /api/members/:id/habits?weekId=18 · returns habits + sticker progress array per day
SELECT … FROM habits JOIN habit_members … JOIN habit_stickers … WHERE tenant_id = $1 AND member_id = $2
Lock / close week
FHS-backlog (week-close)
WeekCloseSummary modal → admin selects disposition (claim/save/invest) → closeWeek() mutation
POST /api/weeks/:id/close · body: {action, rewardName?} · idempotent — 409 if already finalized
UPDATE weeks SET is_finalized=true, closure_snapshot=… + INSERT INTO week_actions (tenant_id, …)
Push nudge (mobile)
FHS-backlog (notifications)
PWA service worker receives push → renders system notification with quick-action buttons (Done / Not yet)
Scheduled job (Railway CRON) calls Supabase push endpoint per member per pending habit after Fajr time
SELECT habits … WHERE NOT EXISTS (sticker today) scoped by tenant_id; result drives notification payload
What's Already Built vs Backlog Gap
✅ Already Shipped
  • habits, weeks, habit_stickers, week_actions schema with tenant_id cascade — FHS-3
  • Drizzle migration committed; schema pushed to staging Postgres
  • RLS policies on all four tables — tenant isolation enforced at DB layer
  • Integration tests (Cucumber + Vitest) covering habit CRUD and tenant isolation scenarios — FHS-218
  • Week finalization model (is_finalized, closure_snapshot, carried_over_stickers)
  • Sticker economy fields (sticker_value, is_allocated) and savings / investments tables
🚧 Backlog Gap
  • Habit CRUD endpoints (POST/PATCH/DELETE /api/habits) — not yet built
  • habit_members junction table (per-member habit assignment) — schema extension needed
  • Habit Tracker UI components (DayCell, HabitForm, WeekNav, MemberHabitsPage)
  • Week-close ritual UI (summary modal + disposition picker)
  • PWA push notifications (daily Fajr / habit nudge via Railway CRON + Supabase)
  • Sticker animation on quick-mark (CSS keyframe — no JS required)
  • Streak computation query and API endpoint
  • Heatmap / analytics view for habit completion history
