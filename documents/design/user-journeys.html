User Journey Map

Family Hub — Multi-Tenant SaaS Platform

4 personas  ·  10 stages each

This map traces how each of Family Hub's four primary personas moves from first awareness through paid retention — and where they might leave. Each stage card shows the trigger, user action, system response, emotional state, and the Jira ticket(s) that build it. Stages without a ticket are flagged as backlog gap — these are the planning holes that need sprint assignment before that persona's journey is fully covered by working software.

Persona 4 — Gaps & Open Questions

  • ▶ Stage 2 (Evaluate): The invite-aware signup page is the most important first impression for this persona. It must show the inviter's name and family context. Not designed or ticketed.
  • ▶ Stage 3 (Sign Up): Invite token → family_members.auth_user_id linkage is architecturally critical. Not confirmed as an explicit child ticket under FHS-90.
  • ▶ Stage 4 (Onboard): There is no invitee-specific first-login experience. The primary user's setup wizard is irrelevant to someone joining an existing family.
  • ▶ Stage 5 (First Value): Both push notifications and the tasks feature are unticketed. This stage cannot happen without them.
  • ▶ Stage 7 (Invite): Role permission errors need a clear UX path ("ask your family admin"). Currently undefined.
  • ▶ Stage 10 (Churn): Family-member inactivity is a silent churn signal for the payer. No detection or nudge flow exists.

Cross-Cutting Risks

System-level risks that cut across all personas and journey stages. Each is tagged with the epic or ADR that governs it.

🔒

Auth session expiry in mid-flow

A Supabase JWT expires while the user is mid-onboarding wizard or mid-checkout. They lose context, land on /login, and see a blank state. No restore point.

FHS-178 FHS-191 Session recovery gap
🔐

Tenant data cross-contamination

A bug in RLS policy or tenant middleware serves Family A's data to Family B. Catastrophic trust failure. Only one family affected in a breach, but all families churn from perception.

FHS-1 FHS-12 FHS-218 ADR-0001
💳

Stripe webhook delivery failure

User pays via Stripe Checkout but the webhook fails. Tenant remains on free tier. User sees "upgrade" prompts on a feature they just paid for. High-urgency support ticket guaranteed.

FHS-73 FHS-78 ADR-0004 idempotency gap
📧

Invite race condition

Two family members accept the same invite link concurrently (e.g. a shared email thread). Two auth_user_id values attempt to populate the same family_members row. Unique constraint race on family_members.

FHS-90 FHS-97 invite idempotency gap
🌐

Subdomain slug collision

Two families claim the same slug concurrently (e.g. "smith-family"). First write wins, second gets a cryptic error. Onboarding abandonment spike.

FHS-23 FHS-29 ADR-0002 collision UX gap
⚠

Schema drift (API vs DB)

A migration ships without updating the Drizzle schema, or vice versa. API returns 500s for a tenant-scoped query. Silent data corruption possible before detection.

FHS-194 FHS-197 ADR-0005
👤

Child data privacy (COPPA / GDPR-K)

Children under the local age minimum may not have their own logins, but their data (maths performance, calendar events, photos) is stored. Parental consent, data ownership, and deletion rights are not yet formally designed.

FHS-1 compliance gap no COPPA ticket
📱

Mobile-first invitees on a web-only app

Persona 4 (invitees) receive invite emails on mobile. The web app is responsive but not a native app yet. Invite-to-first-use friction on iOS Safari (no push, no home-screen install prompt) may cause immediate abandonment.

PWA gap native app gap no ticket
🧙

White-label branding breaks accessibility

A community admin uploads their org's brand colours (e.g. a mosque with dark green on black). The system applies them site-wide, violating WCAG 4.5:1 contrast ratios. Accessibility regression for their entire member base.

FHS-123 brand validation gap no contrast check ticket
